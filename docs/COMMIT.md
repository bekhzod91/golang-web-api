# COMMIT

## Branching

Use short, clear branch names by intent, for example:

- `feat/<topic>`
- `fix/<topic>`
- `docs/<topic>`
- `chore/<topic>`

## Commit Messages

Prefer conventional style:

- `feat: ...`
- `fix: ...`
- `docs: ...`
- `chore: ...`
- `refactor: ...`
- `test: ...`

Keep commits focused and atomic.

## Pull Requests

Each PR should include:

- concise summary of what changed
- reason/impact
- notes about migrations, env changes, or breaking changes
- test evidence (what was run)

If docs or contracts changed, include related updates in the same PR.
